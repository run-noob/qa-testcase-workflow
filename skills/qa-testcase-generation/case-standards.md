# 测试用例编写标准

## 1. 用例编号规则
### 1.1 编号格式
`{MODULE}-{SUB_MODULE}-{TYPE}-{SEQ}`

### 1.2 字段说明
- `MODULE`：模块缩写，如 UM（用户管理）、OS（订单系统）、PAY（支付）
- `SUB_MODULE`：子模块或功能缩写，如 LOGIN、REFUND、PROFILE
- `TYPE`：测试类型缩写
- `SEQ`：三位数字序号，如 001、002

### 1.3 类型缩写
| 类型 | 缩写 |
|------|------|
| 功能测试 | FT |
| 边界测试 | BT |
| 异常测试 | ET |
| 性能测试 | PT |
| 安全测试 | ST |
| 兼容性测试 | CT |

## 2. 标题与结构要求
### 2.1 用例标题
- 标题必须具体，直接描述验证目标
- 避免使用“测试登录”“验证退款”这类空泛表述
- 推荐格式：`[动作] + [对象] + [条件/场景]`

### 2.2 前置条件
- 只保留当前用例执行所必需的条件
- 使用有序列表描述
- 不要混入测试步骤中的动作

### 2.3 测试步骤
- 每一步只写一个核心动作
- 步骤必须可执行、可复现
- 涉及输入时优先写明确值，不写“合法数据”“非法数据”

### 2.4 预期结果
- 结果必须具体且可验证
- 尽量写清页面反馈、接口返回、状态变化、数据库变化等
- 避免仅写“正常”“成功”“失败”

### 2.5 测试数据
- 优先给具体值与说明
- 敏感数据使用脱敏示例
- 数据项需与步骤和预期保持一致

## 3. 设计原则
- 一条用例只验证一个清晰测试点
- 主流程、异常流、边界流应拆分为独立用例
- 高风险规则优先单独成例，避免与低风险点混合
- 对于可穷举的规则组合，优先使用等价类、边界值、判定表等方法控制数量

## 4. 质量校验清单
- 用例 ID 是否符合规则
- 标题是否清晰、无歧义
- 步骤是否可独立执行
- 预期是否可验证
- 功能点是否可追溯
- 是否存在明显冗余或多目标混测
