# 测试用例优先级划分规则

## 1. 优先级定义
### P0
适用于核心主链路、阻塞性功能、上线失败会直接影响核心业务可用性的场景。

典型场景：
- 登录、注册、下单、支付、退款主流程
- 关键状态流转
- 资金、权限、关键数据一致性相关能力

### P1
适用于重要业务分支、关键流程功能点验证、失败后会严重影响用户体验但存在替代路径的场景。

典型场景：
- 核心流程的重要分支
- 关键异常处理
- 高频使用的重要功能点

### P2
适用于一般功能、常见边界、次要业务分支。

典型场景：
- 普通表单校验
- 一般异常提示
- 非核心但常用功能

### P3
适用于边缘场景、极端条件、低频兼容性或体验细节。

典型场景：
- 极端输入
- 罕见异常链路
- 非阻塞性的兼容性细节

## 2. 判定原则
- 涉及资金、权限、隐私数据的用例，优先级通常不低于 P1
- 新增功能的核心验收链路，优先级通常不低于 P1
- 回归时命中过往高风险缺陷区域的用例，可上调一级
- P0 应保持克制，只覆盖真正阻塞上线的关键路径

## 3. 分布建议
| 优先级 | 建议占比 |
|--------|----------|
| P0 | 10% - 20% |
| P1 | 30% - 40% |
| P2 | 30% - 40% |
| P3 | 10% - 20% |
