# QA 变更影响分析 Skill

## 适用场景
当用户要求做变更分析、影响范围分析、回归范围识别、存量用例差异评估时使用本 Skill。

建议调用示例：
- `/qa-change-diff`
- `/qa-change-diff 退款需求`
- `/qa-change-diff prd/current/output/refund-analysis.md`

> 💡 本 Skill 也可以作为 `/qa-workflow` 工作流的一部分自动执行。如需执行完整流程（PRD分析 → 变更分析 → 用例生成 → 评审 → 合并归档），建议使用 `/qa-workflow` 一键执行。

## 输入参数
- `$ARGUMENTS` 可选。
- 可传需求名、`*-analysis.md` 路径，或目标模块提示。
- 未传参数时，默认从 `prd/current/output/` 中定位最新或唯一的 `*-analysis.md`。
- 如果无法唯一定位分析报告，先向用户确认。

## 前置条件
优先检查：
1. `prd/current/output/` 下存在 `*-analysis.md`。
2. `test-cases/index.md` 存在并可读取。

如果 `test-cases/index.md` 不存在、为空、或全量用例库尚未建立：
- 不要阻塞执行。
- 在报告中明确标记：`当前存量用例库为空，本次按新增需求规划测试用例，不做存量复用/修改分析。`

## 强制规则
1. 任务开始前先读取 `glossary/` 下所有术语文件。
2. 优先参考 `skills/qa-change-diff/change-diff-template.md` 模板。
3. 必须先读取 `test-cases/index.md` 获取全局索引。
4. 不要一次性读取整个 `test-cases/` 目录。
5. 只按需读取与本次需求相关的模块用例文件。
6. 如果相关模块过多，按“直接修改模块 → 强关联模块 → 间接影响模块”分批分析。
7. 输出写入 `prd/current/output/{feature-name}-change-diff.md`。
8. 本阶段只做分析，不直接修改全量用例库。

## 执行流程

### Step 1：加载上下文
读取：
- `glossary/*.md`
- `prd/current/output/*-analysis.md`
- `test-cases/index.md`
- 与涉及模块相关的模块级 `index.md` 或具体用例文件（按需）

### Step 2：识别影响模块
基于分析报告中的“涉及模块清单”和功能点，输出模块影响分层：
- 直接变更模块
- 关联影响模块
- 间接影响模块

每个模块需给出影响原因，例如：
- 业务规则调整
- UI 流程变更
- 接口字段变更
- 状态流转变更
- 权限/角色变更
- 上下游联动变更

### Step 3：建立存量用例映射
对于每个相关模块：
1. 从 `test-cases/index.md` 定位对应文件。
2. 按需读取文件中的用例标题、ID、核心步骤、预期结果摘要。
3. 建立“功能点 → 存量用例”映射。

### Step 4：识别变更点类型
对每个功能点判断：
- 新增功能
- 修改功能
- 废弃功能
- 不变功能

并明确变化类型，如：
- 规则变化
- 页面交互变化
- 接口入参/出参变化
- 状态机变化
- 权限变化
- 数据结构变化

### Step 5：评估存量用例影响
对已存在用例逐条给出评估结论：
- 可直接复用
- 需要修改
- 需要废弃
- 无法覆盖，需要新增

对“需要修改”的用例，必须说明至少一项具体修改点，例如：
- 步骤需更新
- 预期结果需更新
- 前置条件变化
- 测试数据变化
- 优先级变化

### Step 6：给出回归范围建议
输出三层回归范围：
- 必须回归（P0）
- 建议回归（P1）
- 可选回归（P2）

回归建议应结合：
- 直接受影响功能
- 强依赖模块
- 历史高风险区域
- 核心业务主链路

### Step 7：输出变更分析报告
写入：`prd/current/output/{feature-name}-change-diff.md`

报告建议结构：

```markdown
# {需求名称} - 变更影响分析报告

## 1. 分析概览
## 2. 模块影响总览
## 3. 详细变更分析
## 4. 新增用例规划
## 5. 回归测试建议
## 6. 风险提示
```

报告中至少应包含：
- 影响模块统计
- 模块影响总览表
- 变更点清单
- 受影响存量用例清单
- 新增用例规划
- 回归建议
- 风险提示

## 上下文控制要求
- 同时加载的模块尽量不超过 3 个。
- 如果单模块用例过多，先读取 ID/标题清单，再按需补读详细内容。
- 分析完一个模块后，用摘要替代其全文，继续下一个模块。

## 完成回报
完成后向用户反馈：
- 报告路径
- 影响模块数
- 涉及存量用例数
- 可复用用例数
- 需修改用例数
- 需废弃用例数
- 计划新增用例方向数
- 是否建议进入下一步 `/qa-testcase-generation`

## 质量要求
- 不能只说“有影响”，必须说明“为什么有影响”。
- 不能只说“需要修改用例”，必须指出修改维度。
- 不要把“潜在影响”混写成“确定影响”，要区分清楚。
- 如果当前项目为新项目或空用例库，要明确说明，而不是伪造复用关系。

## 失败处理
如遇以下情况，优先说明并请求确认：
- 未找到分析报告
- 模块归属不明确
- 用例索引缺失且用户未确认按新项目处理
- 相关模块过多且无法确定分析优先级
- 存量用例结构过于混乱，无法稳定定位对应关系
