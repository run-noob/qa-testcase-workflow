# QA 测试用例评审 Skill

## 适用场景
当用户要求对已生成的测试用例进行质量评审、自检、覆盖检查、规范校验时使用本 Skill。

建议调用示例：
- `/qa-testcase-review`
- `/qa-testcase-review 退款需求`
- `/qa-testcase-review prd/current/output/test-cases/test-case-summary.md`

> 💡 本 Skill 也可以作为 `/qa-workflow` 工作流的一部分自动执行。如需执行完整流程（PRD分析 → 变更分析 → 用例生成 → 评审 → 合并归档），建议使用 `/qa-workflow` 一键执行。

## 输入参数
- `$ARGUMENTS` 可选。
- 未传参数时，默认评审 `prd/current/output/test-cases/` 下当前需求的所有产出文件。

## 强制规则
1. 任务开始前先读取 `glossary/`相关术语文件。
2. 必须读取 `prd/current/output/*-analysis.md`，必要时读取 `*-change-diff.md`。
3. 必须读取 `prd/current/output/test-cases/test-case-summary.md`。
4. 逐个读取当前需求下的生成用例文件进行评审。
5. 默认只输出评审报告，不直接修改用例文件。
6. 如果需要自动修复，必须先得到用户明确同意。
7. 评审产出写入 `prd/current/output/test-cases/review-report.md`。

## 执行流程

### Step 1：加载评审材料
读取：
- `prd/current/output/*-analysis.md`
- `prd/current/output/*-change-diff.md`（如存在）
- `prd/current/output/test-cases/test-case-summary.md`
- `prd/current/output/test-cases/` 下所有当前需求生成的用例文件

### Step 2：多维度评审

#### 2.1 完整性评审
检查：
- 是否覆盖所有功能点
- 是否覆盖核心主流程
- 是否覆盖关键反向/异常场景
- 是否覆盖边界条件
- 是否覆盖必要的跨模块交互场景
- 是否存在明显缺失场景

#### 2.2 准确性评审
检查：
- 步骤是否与 PRD 一致
- 预期结果是否一致
- 前置条件是否正确且充分
- 测试数据是否合理有效
- 用例之间是否存在逻辑冲突

#### 2.3 规范性评审
检查：
- 用例 ID 是否符合编号规则
- 优先级分配是否合理
- 文件格式是否符合模板要求
- 描述是否清晰无歧义
- 步骤是否可独立执行
- 预期是否可验证

#### 2.4 设计质量评审
检查：
- 是否合理应用等价类、边界值等设计方法
- 是否存在可合并的明显冗余用例
- 是否存在“一条用例覆盖多个目标”的问题
- 是否存在高风险场景优先级过低的问题

### Step 3：生成评审结论
将问题归类为：
- 严重问题：必须修改，否则不建议通过
- 一般问题：建议修改，影响质量但不致命
- 优化建议：可后续优化
- 缺失场景：建议新增
- 冗余用例：建议删除或合并

### Step 4：输出评审报告
写入：`prd/current/output/test-cases/review-report.md`

建议结构：

```markdown
# {需求名称} - 用例评审报告

## 1. 评审概览
## 2. 评审评分
## 3. 问题清单
## 4. 缺失场景
## 5. 冗余用例
## 6. 评审结论
```

报告中至少包含：
- 用例总数
- 通过用例数/需修改用例数
- 建议新增用例数
- 建议删除或合并用例数
- 完整性/准确性/规范性/设计质量评分
- 具体问题清单
- 最终结论：通过 / 有条件通过 / 不通过

### Step 5：可选自动修复
如果用户明确同意自动修复，则仅允许修复以下问题：
- 格式问题
- 编号或优先级明显错误
- 缺失的轻量字段
- 明显不准确的预期结果表述
- 少量缺失场景补充

修复完成后：
- 更新相关用例文件
- 更新 `test-case-summary.md`
- 在评审报告末尾追加“自动修复记录”

## 完成回报
完成后向用户反馈：
- 评审报告路径
- 用例总数
- 严重问题数
- 一般问题数
- 缺失场景数
- 冗余用例数
- 最终评审结论
- 若结论通过，是否建议进入下一步 `/qa-testcase-merge`

## 质量要求
- 不能只说“覆盖不足”，必须指出缺什么。
- 不能只说“描述不清”，必须指出哪一部分不清。
- 不能把风格建议混同为严重缺陷。
- 对每个严重问题，都应给出具体修正建议。
- 评审结论必须与问题清单一致，不能前后矛盾。

## 失败处理
如遇以下情况，优先说明原因：
- 测试用例文件缺失
- 汇总文件不存在
- 分析报告与用例内容明显不匹配
- 评审范围无法确定
- 当前材料不足以完成有效评审
